Paclet and package structure

~/
    PacletInfo.wl
    Kernel/
        p.wl
        p1.wl
        ...
    Workbench/
        p.nb
        ...
    ...
PacletInfo.wl
PacletObject[
    <|
        "Name" -> "p",
        "Version" -> "0.0.1",
        "WolframVersion" -> "13+",
        "Extensions" ->
            {
                {
                    "Kernel",
                    (*the root path of packages*)
                    "Root" -> "Kernel",
                    (*the (sub)contexts*)
                    "Context" -> 
                        {
                            "p`",
                            "p`p1`"
                        }
                }
            }
    |>
]
p.wl
(*force to reload packages if necessary.*)
Get["p`p1`"]

(*set the context as p` and load the subcontext p`p1`.*)
BeginPackage["p`",{"p`p1`"}];

(*clear the definitions when reloading the package if necessary.*)
Unprotect@@Names["p`*"];
ClearAll@@Names["p`*"];

(*declare public symbols.*)
foo::usage = "";

(*set the private context.*)
Begin["`Private`"];

(*define the symbols.*)
foo = Identity;

(*end the private context and revert to p`.*)
End[];

(*protect the public symbols if necessary.*)
Protect@@Names["p`*"];

(*end the context, and add p` to $ContextPath.*)
EndPackage[];

Symbol types

  • Constant - storing global parameters.

  • Association - storing object-like data.

  • Questing function - defining the type of variables. The definition should be complete.

  • Main function - implementint the functionalities of the package. To prevent unexpected evaluations, define it when it needs to be defined.

  • Helper function - assisting the main function. The dependency should respect a top-down tree-like structure;

  • Formatting function - formatting the output. It's better to separate the intermediate values passed between functions and the return values.

Naming conventions

  • The body functions with similar functionality are named with the same prefix.

  • The questing functions are named with the suffix *Q.

  • The helper functions are named by specifying the subcontext foo`foo1.

  • The local copy of the variable foo is named as $$foo.

  • The name of local variables should reveal the types, e.g. repeated patterns foos__, list pattern fooList_.

Some tips

Subcontexts

The p1.wl file with subcontext p`p1` should be placed adjacent to the p.wl file, see paclets and paclet development.

When there are deeper levels of subcontexts such as p`p1`p11, the p11.wl file should be placed in a ~/Kernel/p1/ directory, with the subcontext hierarchy corresponding to the directory hierarchy. The first level of the subcontext hierarchy, p`, is free and is mapped to the ~/Kernel/ directory itself.

Old-styled init.wl files

Older packages use a special workaround allowing the context p` to be mapped to a p.wl file in a p/ directory found in $Path.

p/
    p.wl
    Kernel/
        init.wl
    ...

If a context is mapped to a location in $Path, but there are no .wl files with appropriate names found there, the kernel searches for an init.wl file in the p/Kernel/ subdirectory. The init.wl file includes a command such as Get["p`p`"], where the double context assists in mapping to a p.wl file in the p/ root directory.

In the new-styled paclet, this is supported but strongly discouraged.

Old-versioned auto-completion

In the old versions of Mathematica, the auto-completion with sub-package is problematic: under the parent package p`, the public functions in the sub-package p`p1` are not automatically completed, see sub-packages and auto-completion. The solution is to decouple the parent package and rename it as p0`, then load the sub-package p`p1` in p0`.
